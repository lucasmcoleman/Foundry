# Audit Report — Foundry

**Identity:** `github.com/lucasmcoleman/pipeline` (repo dir: `/server/programming/Foundry`) — a **user-owned** 5/6-stage LLM fine-tuning and hybrid-quantization pipeline (QLoRA training, LoRA merge/export, heretic abliteration, REAP MoE pruning, MagicQuant hybrid GGUF, AMD Quark/OGA ONNX INT4 AWQ, and HF Hub upload) with a CLI and a FastAPI web UI, targeting AMD APU unified-memory hardware (Strix Halo gfx1151, 124 GB shared RAM).

**Health grade: B** — The hardware-specific core (subprocess isolation + custom shard-streaming loaders) is genuinely well-engineered and most prior CRITICAL bugs are verifiably fixed at HEAD; the grade is held back by one structurally important duplicated CLI path, a residual silent-corruption skip-logic gap, and broadly stale documentation.

---

## Executive Summary

Foundry is a mature, single-operator ML pipeline whose architecture is well-matched to its real constraint — reclaiming GPU/GTT memory between stages on a unified-memory APU. Its defining strengths (per-stage subprocess isolation, the custom `fast_train_zeroclaw`/`fast_export` shard-by-shard loaders, atomic per-shard merge writes) are deliberate and elegant. Crucially, **the bundled audit docs (`AUDIT_REPORT.md`, `LOGIC_AUDIT.md`, `HANDOFF.md`, `OPEN_QUESTIONS.md`) and the recon notes are STALE**: all three previously-CRITICAL bugs — repr()/!r f-string injection, an always-`True` `validate_dataset`, and the `did_training` model-card flag — are already fixed in HEAD `3c9bd03` (verified directly: `pipeline.py:358` returns `all_ok`, `pipeline.py:1309` reads `"training" in _enabled`, and grep finds zero bare interpolations). Overall health is good. The single most important thing to fix is **incomplete consolidation**: a `services.py` service layer was extracted and the UI delegates to it, but the CLI (`core/pipeline.py`) was never migrated and still carries a full ~1500-line parallel implementation that has measurably diverged — so the same config produces different adapters through CLI vs UI, and every fix must be made twice.

---

## What's Right

- **Subprocess-per-stage isolation is the correct core abstraction.** Each stage runs as its own process, so GPU/GTT memory is fully reclaimed between stages on the unified-memory APU — a documented, deliberate decision, not accidental.
- **The custom fast loaders are the elegant heart of the project.** `fast_train_zeroclaw.fast_load_quantized_model` and `fast_export.streaming_merge` use a meta-device skeleton + shard-by-shard load + inline BnB quant to keep peak memory ~30 GB (train) / ~6 GB (merge) vs 80 GB+ for naive HF loading. Clean, importable, well-commented.
- **Atomic per-shard merge writes** (`fast_export.py:235-238`): each shard is written to a `.tmp` file then renamed, so a crash mid-write cannot leave a corrupt shard.
- **The injection class is genuinely closed.** Every generated subprocess script in `core/pipeline.py` and `core/services.py` interpolates config string values via `repr()`/`!r` (training, export, heretic, reap, magicquant, upload, and the worktree ONNX service), so f-string breakout from model names/paths/dataset IDs is not exploitable.
- **No dangerous primitives and no secrets.** Zero `eval`/`exec`/`pickle.load`/`yaml.load`/`os.system`/`shell=True`; YAML uses `safe_load`; all subprocesses use argv lists. No hardcoded tokens; `.env` gitignored; HF token sourced only from env or `~/.cache/huggingface/token`; tracked `ui/config.json` holds only a username.
- **Appropriate web defenses for the threat model.** Path-traversal regexes on `/api/runs/{model}/{logfile}` and workflow names; an `asyncio.Lock` + re-check closes the `/api/run` double-launch race; consistent Bearer/token auth on every route *when enabled*; Docker image runs as non-root `foundry` user.
- **Real packaging and dependency hygiene now exist.** `pyproject.toml` with declarative deps + optional extras, `tenacity` exponential-backoff retry on HF API calls, namespace validation before repo creation, editable installs replacing the old fragile symlink web, and `pydantic-settings`/`.env` support.
- **`onnx_quark.py` (worktree) is the model the rest of the code should follow:** a real importable module with typed functions where the subprocess script is a ~10-line shim — proof the team knows the better pattern.
- **Strict, consistent stage pattern** (dataclass + Pydantic mirror + `Service.build_script` + `stage_*`/`do_*` + `STAGES` registration) makes new stages low-risk near-clones, and the `stage_upload_dry_run` credential/repo/model-card validation is good DX before a multi-GB push.

## What's Wrong

### Critical
*None.* The three issues the bundled docs flag as CRITICAL (f-string injection, always-`True` `validate_dataset`, `did_training` flag) are all already fixed at HEAD `3c9bd03` and verified. Treat any tool/doc still reporting them as stale.

### High

- **CLI keeps a full divergent parallel implementation of every stage** — `core/pipeline.py:406-965` vs `core/services.py:65-575` (pipeline.py does **not** import services.py; verified). The two copies have measurably diverged: CLI uses `prepare_model_for_kbit_training` while services uses hand-rolled kbit prep with `requires_grad=False` + fp32 norm upcast; CLI uses `warmup_ratio=0.05` (`pipeline.py:515`) while services uses `warmup_steps` (`services.py:245`); services supports `use_rslora`/`packing`, CLI does not; CLI has GPT-OSS `<|channel|>analysis` handling services lacks; defaults differ (`max_seq_length` 8192 vs 4096, `optim` `adamw_8bit` vs `paged_adamw_8bit`). **Impact:** the same config yields different adapters via CLI vs UI; every bugfix must be done twice and silently drifts. This is the root cause spawning most other findings. **Fix:** make `pipeline.py` stage functions call the same Service classes the UI uses; add a test asserting CLI and UI generate equivalent scripts.
- **Code-generation-as-execution is the core abstraction; stage bodies are Python emitted as escaped strings** — `core/pipeline.py:425-542, 661-950, 1054-1166`; `core/services.py` (~531 f-string lines). **Impact:** no IDE/lint/type/breakpoint support; tracebacks point at anonymous `_stage_*.py` temp files; every quoting mistake is a latent runtime bug (the entire injection class existed only because of this). Maintenance surface grows linearly as stages are added. **Fix:** move stage bodies into real importable modules (as `fast_train_zeroclaw.py`/`fast_export.py`/`onnx_quark.py` already do) and reduce the subprocess wrapper to a fixed launcher reading a JSON/dataclass config.
- **Web UI binds `0.0.0.0` with authentication OFF by default** — `ui/app.py:38` (`API_KEY = os.environ.get("FOUNDRY_API_KEY", "")`), `ui/app.py:1202` (`host="0.0.0.0"`), `run.sh`, `Dockerfile`; `verify_api_key()` is a no-op when the key is empty; `/` and `/health` are unauthenticated. **Impact:** out of the box, anyone who can route to port 7865 on the LAN can drive the full pipeline; given the user's documented SWAG+Authelia reverse-proxy infra, accidental external exposure is plausible. **Fix:** default to `127.0.0.1` (override to `0.0.0.0` only via explicit env var), or refuse to bind a non-loopback interface without an API key; at minimum emit a loud startup warning.
- **Primary test suite is broken; core logic has zero offline coverage** — `tests/test_pipeline.sh:19` hardcodes `PIPELINE_DIR="/server/programming/pipeline"`, a path that no longer exists (verified). All 10 tests fail (`activate_env` sources a venv from the dead path); the only pytest module needs a GPU and a 5 GB download; `make test` fails outside the venv. **Fix:** derive `PIPELINE_DIR` from the script's own dirname, port the worktree mock-based offline pattern to master, and mark the GPU test with a pytest marker.

### Medium

- **`/api/run` executes attacker-controlled Python subprocesses by design (RCE if auth is off)** — `ui/app.py:1137-1156`, `run_script()` at `ui/app.py:182-217`. `repr()` prevents f-string breakout, but operator-supplied `model_name`/`source_model`/`datasets`/`llamacpp_path` flow into `load_dataset`, `snapshot_download`, `sys.path.insert`, and `git clone`/`pip install`/`cmake`. **Impact:** when auth is off (default) and the port is reachable, a caller gets shell-equivalent capability. The mitigation is access control, not sanitization. **Fix:** require the API key (the dependency exists; the key is just empty by default); never expose the UI to an untrusted network; document that running the UI grants shell-equivalent access. *(Weighting note: kept Medium rather than High — its severity is a direct multiplier of the `0.0.0.0`+optional-auth High finding above and adds no independent injection bug; fixing the bind/auth default neutralizes it.)*
- **Stage skip/resume keys on artifact existence, not completion — silent reuse of truncated artifacts** — `ui/app.py:393-398, 447-462, 528-533, 683-691`; CLI `pipeline.py` has *no* skip logic at all. PEFT writes `adapter_config.json` early in `save_pretrained`, so a crash after config-write but before `adapter_model.safetensors` leaves a dir that passes the skip; export skips on any `merged_model/*.safetensors`; magicquant on any `*.gguf`. The code documents this but no fix exists. **Impact:** a crashed run is silently "resumed," propagating a truncated/stale artifact downstream (e.g. quantizing a partial merge) and producing a silently wrong final model. This is the **highest-value residual correctness gap.** (Atomic per-shard writes mitigate shard-level corruption but there is no whole-stage marker.) **Fix:** write `_stage_complete.json` (timestamp + config hash) only after the subprocess returns 0 AND the key file exists non-empty; gate skips on the marker; share the logic between CLI and UI.
- **Broken `foundry` console entrypoint** — `pyproject.toml:70` declares `foundry = "core.pipeline:main"`, but `core/pipeline.py` has no `main()` (CLI lives under `if __name__ == "__main__":` at `pipeline.py:1430`; verified). **Impact:** after the documented `pip install -e .`, running `foundry` fails with `AttributeError`. **Fix:** refactor the `__main__` block into `def main()` and call it from the guard — also makes the CLI importable/testable.
- **`logging_config.py` (structlog) is dead code, never imported** — verified zero importers across `core/`/`ui/`. Meanwhile `pipeline.py:183-188`, `hf_upload.py:26-29`, and `onnx_quark.py:31-35` each define their own `LogFn`/`_default_log`. The "structured logging resolves CQ-2" claim is inaccurate. **Fix:** either wire `configure_logging()`/`ws_callback_factory` in at startup and delete the duplicate triples, or delete `logging_config.py` and drop the structlog dep.
- **Config fragmented across three incompatible schemas; YAML loader silently ignores most of it** — `core/config.py:22-84` (FoundrySettings, used only for a port number), `pipeline.py:27-146` (dataclasses), `ui/app.py:113-177` (Pydantic). Defaults conflict (`optim`, `max_seq_length`). The CLI loader reads only `data["training"]` (`pipeline.py:1462-1464`), and `configs/default.yaml` uses a flat top-level schema with no `training:` wrapper — so the file literally named *default* is a silent no-op via `--config`. **Fix:** promote one Pydantic model into a shared `core/config` module imported by both entrypoints; make the loader populate all sections; fix `default.yaml` to match.
- **UI training path silently ignores `warmup_ratio`; always uses `warmup_steps=10`** — `ui/app.py:127`, `services.py:245`. CLI uses `warmup_ratio=0.05` (`pipeline.py:515`). **Impact:** the same config yields different LR schedules; for large datasets 10 fixed steps ≠ 5% warmup, changing convergence silently and making UI results non-reproducible from CLI. **Fix:** unify on one warmup parameterization and log the effective warmup. *(This is a concrete symptom of the High CLI-divergence finding.)*
- **ONNX-only re-run keys its skip on a GGUF artifact this path never creates** — `.worktrees/quark-onnx/ui/app.py:461-476`. `do_export` derives its skip from `mq_enabled` only; with ONNX enabled but MagicQuant off, it checks for `merged_model/model-bf16.gguf`, which nothing in core ever writes (export only emits safetensors). **Impact:** redundant full re-merge of a 30B+ model on every ONNX-only re-run (output stays correct). **Fix:** compute `safetensors_needed = mq or onnx or heretic or reap` and check `merged_model/*.safetensors`; fall back to the gguf check only for a pure GGUF-export run.
- **Auto-clone/auto-install of llama.cpp and amd/Quark without pinned SHAs or wheel hashes** — `core/services.py:746-790`, `core/pipeline.py:363-396`, `.worktrees/quark-onnx/core/onnx_quark.py:38-101`. `git clone` + `cmake` for llama.cpp; `pip install amd-quark>=0.11 onnxruntime-genai>=0.6` and `git clone --branch release/0.11 .../Quark.git` then `runpy` of `quantize_quark.py`. **Impact:** supply-chain exposure (force-push/typosquat → arbitrary code in the venv); URLs are legit so not directly attacker-triggerable. **Fix:** pin clones to commit SHAs and verify; pin the wheels to exact versions/hashes — also improves reproducibility.
- **API key compared with non-constant-time `!=`** — `ui/app.py:44` and `ui/app.py:1186`. Theoretical timing side-channel; impractical on a jittery LAN but trivial to fix. **Fix:** use `hmac.compare_digest()` for both REST and WebSocket, and make the secret-handling consistent (REST compares against `"Bearer " + KEY`, WS against raw `KEY`).

### Low

- **Heretic Pareto `best_trials` loop is dead; best is always `sorted_trials[0]`** — `pipeline.py:912-925`, `services.py:544-552`. The KL threshold the comment implies does not exist. No wrong output; misleads readers. **Fix:** use `best = sorted_trials[0]` or implement a real KL acceptance threshold.
- **`detect_response_template()` is dead code (LOGIC_AUDIT A6 is moot)** — `fast_train_zeroclaw.py:303-343`; never called. All paths set `completion_only_loss=True` and use no `DataCollatorForCompletionOnlyLM`. A6's "lstrip causes zero/full loss" describes a function that never runs. **Fix:** remove it (or wire it in deliberately); downgrade A6 to dead code.
- **Composite-model detection and quant `skip_names` rely on single substring signals** — `fast_train_zeroclaw.py:75, 157-158`. `_is_composite = hasattr(config,'text_config')` remaps without confirming `model.language_model.*` keys (unlike the safer `fast_export.py:191` check); `skip_names` substring-matches `embed`/`norm` so `normalize_*`/`embedding_projection` is left unquantized. Mis-detection is severe-if-hit but unlikely for Qwen/Llama; `skip_names` over-match only wastes memory. **Fix:** require `any(k.startswith('model.language_model.'))` before remapping; match the leaf module name / `isinstance(LayerNorm/RMSNorm)` for skips.
- **REAP arch allow-list mixes class names with HF repo-id strings that can never match** — `pipeline.py:976-987`, `ui/app.py:568-579`. `_detect_model_arch` returns `architectures[0]` (a class name), but the list includes `'Qwen3-Coder-30B-A3B-Instruct'`/`'gpt-oss-20b'` (unreachable), so a real gpt-oss model would be silently skipped. **Fix:** populate with actual `architectures[0]` class strings cross-checked against `reap.model_util.MODEL_ATTRS`.
- **REAP integration hardcodes an absolute sibling path and stubs 14 deps inline** — `services.py:646`, `pipeline.py:1086` both do `sys.path.insert(0,'/server/programming/reap/src')` and inject 14 fake modules (vllm, lm_eval, deepspeed, …). REAP is not in `pyproject.toml`. **Impact:** non-portable; a 15th heavy dep silently breaks the stage. **Fix:** make the REAP path configurable, document it as an optional external integration, and centralize the stub block (currently duplicated verbatim).
- **`fast_load` fabricates a 2-entry `hf_device_map` and sets `is_quantized=False` to bypass HF validation** — `fast_train_zeroclaw.py:296-298`; `'_dummy'` is a synthetic key. A working but brittle hack against transformers/accelerate internals. **Fix:** pin/comment the validated transformers+accelerate versions and add a startup version assertion so an upgrade fails loudly.
- **Stage source-resolution priority chain duplicated 3+ times** — `pipeline.py:1194-1208`, `services.py:799-831`, `ui/app.py:843-866` (and REAP source in `pipeline.py:1016-1021` vs `app.py:617-624`). Currently agree, but adding an intermediate stage means editing all copies. **Fix:** define the artifact-source priority once as data (natural home: the `Artifacts` class, `pipeline.py:151`).
- **`POST /api/config` persists arbitrary unvalidated JSON** — `ui/app.py:992-998` (`cfg.update(body)`). Inert today (only `hf_username` is consumed) but a forward-looking vector. **Fix:** validate against a small Pydantic model / allowlist.
- **HF token injected into every stage subprocess, not just upload** — `ui/app.py:200-204`, `activate.sh:26-28`. Widens blast radius if a stage runs untrusted model code during loading. **Fix:** scope `HF_TOKEN` to the upload stage only.
- **CLI subprocess has no timeout/interrupt** — `pipeline.py:191-211` (`proc.wait()` with no timeout); the UI has a stop button (`os.killpg`, `app.py:1167`) but the CLI has no equivalent. A gfx1151 kernel hang wedges the CLI indefinitely. **Fix:** thread an optional per-stage timeout/deadline through `_run()` and kill the process group on expiry; optionally a stdout-silence heartbeat watchdog. Reliability, not correctness.
- **Training supports only the chat `messages` dataset format** — `fast_train_zeroclaw.py:423-453` (hardcoded `apply_chat_template(ex['messages'])`). No alpaca/`{text}`/`{prompt,completion}` branch. **Fix:** add a format-detection/normalization step or a configurable `dataset_format` field.
- **Generated `_stage_*.py` scripts accumulate uncleaned in output dirs** with embedded config values; only workflow JSON has an unlink path. **Fix:** clean up stage scripts after success (or write to a temp dir).

## What Needs Improvement

- **Consolidation is the throughline.** Almost every High/Medium structural finding (CLI vs UI divergence, warmup split, no CLI skip logic, duplicated source-resolution chains, duplicated `LogFn`/`_default_log`, duplicated REAP stubs) is a symptom of the half-finished service-layer extraction. Putting the CLI on `services.py` and moving stage bodies into importable modules resolves the largest cluster at once.
- **Code-generation pattern caps maintainability.** The program-as-string approach is the root of the injection class and makes 90% of stage logic un-lintable. `onnx_quark.py` already demonstrates the better module-plus-shim pattern; standardize on it.
- **Documentation is broadly stale and actively misleading.** README still says "4-stage" and omits heretic (abliteration) and reap (MoE pruning); README/OPEN_QUESTIONS claim the UI has no auth though Bearer auth shipped in 0.2.0; AUDIT_REPORT/HANDOFF/LOGIC_AUDIT still document the pre-rename `unsloth-env` symlinks and the already-fixed CRITICAL bugs against dead `/server/programming/pipeline` paths. This generates false positives in every downstream review (including the auditors who flagged it here). Date-stamp these as historical or regenerate them; remove the leftover `pipeline.egg-info` duplicate.
- **No CI and a broken test harness.** `make lint`/`make test` and a `dev` extras group exist, but there is no `.github/workflows/`, the `format` target is a placeholder, and `test_pipeline.sh` can't run. Regressions in the fragile script generators are caught only manually.
- **No integrity/identity on reused artifacts** (see skip-logic finding) and **no config-hash gating** mean the same output dir reused across models/configs can false-skip.

## Missing Features & Roadmap

Prioritized by value-to-effort. The `docs/superpowers/specs/` design docs are detailed and implementation-ready, and the strict stage pattern makes new stages low-risk near-clones.

1. **FLM/Q4NX stage (highest value — the user's stated primary deployment).** `docs/superpowers/specs/2026-05-07-flm-q4nx-stage-design.md` is *Approved* and fully specced (`core/flm_q4nx.py`, `FlmConfig`, `FlmService`, UI panel, tests) but has **zero** implementation (verified: grep returns nothing in `core/`/`ui/`). The `.q4nx`/FastFlowLM NPU format the user actually runs cannot be produced today; models ship only as GGUF or (on a branch) ONNX. The spec mirrors the existing ONNX stage. **Land the ONNX worktree first, then implement FLM as the near-clone.**
2. **Merge the ONNX/Quark INT4 AWQ stage to master.** It is feature-complete on `feat/quark-onnx-stage` (worktree `da4c16a`) but not on master, so the default branch cannot produce ONNX/OGA models for Lemonade, and the "5-stage" framing isn't on the default branch. (Fold in the ONNX-only skip-check fix while merging.)
3. **Pre-upload output-quality gate.** No validation exists between merge/quant and `stage_upload` (the merge loop counts tensors but never verifies loadability/completeness). A subtly broken merge or bad quant tier ships to HF and is found by an end user. **Add a lightweight stage: load the model, one short generation, optional perplexity delta; fail before upload.** High value-to-effort given expensive full runs; also auto-catches the known Qwen3.5 empty-content issue.
4. **GPU-memory preflight check.** None exists despite OOM-from-shared-memory being the documented dominant failure (`OPEN_QUESTIONS.md` item 3; CLAUDE.md repeatedly warns to unload LM Studio). Read `torch.cuda.mem_get_info()`/`rocm-smi` at stage start vs a size estimate and abort early with free-vs-needed. Low effort, high daily UX payoff.
5. **Restore or remove `datagen/`/`gardener/`.** No tracked source exists (verified), yet `pyproject.toml:56-62` still ships the `datagen` extras group and CLAUDE.md/AUDIT_REPORT reference them — so `pip install -e .[datagen]` pulls deps for nonexistent modules and contributors hit a dead end. Restore from git history or delete the extras + references and document how training data is now produced.
6. **CI workflow.** Add `.github/workflows/` running `make lint` + CPU-only/mocked `make test` on push/PR; add `ruff` and wire the `format` target.
7. **Broader dataset-format support** (alpaca/instruction, plain text, prompt/completion) and **per-architecture quant calibration tables** (future work per both specs and the user's own memory notes) — lower priority, expand applicability.
8. **Observability beyond log streaming** (GPU memory, stage timings on `/health`) — minor.

## Recommended Next Steps

An ordered, actionable punch-list:

1. **Migrate the CLI onto `services.py`.** Make `pipeline.py` stage functions call the same Service classes the UI uses; port pipeline.py-only behavior (GPT-OSS prefix, kbit approach) into services first so nothing is lost. Add a test asserting CLI and UI generate equivalent scripts. (Resolves the High divergence finding + warmup split + duplicated chains.)
2. **Add `_stage_complete.json` completion markers** (timestamp + config hash), written only after exit 0 AND the key file (e.g. `adapter_model.safetensors`, not just `adapter_config.json`) exists non-empty; gate all skip/resume on the marker, shared between CLI and UI. (Closes the top correctness gap.)
3. **Fix the two breakages that block install/test:** refactor `pipeline.py`'s `__main__` into `def main()` (real `foundry` entrypoint), and make `tests/test_pipeline.sh` derive its dir from the script location (then port the worktree's offline mock pattern and mark the GPU test).
4. **Make auth opt-out:** default-bind `127.0.0.1`; require an explicit env var + API key to bind `0.0.0.0`; loud startup warning otherwise; switch both API-key comparisons to `hmac.compare_digest`.
5. **Refresh/retire docs:** update README to document all six stages (incl. heretic + reap) and the shipped Bearer auth; date-stamp or delete `AUDIT_REPORT.md`/`HANDOFF.md`/`OPEN_QUESTIONS.md`/`LOGIC_AUDIT.md` as historical; remove `pipeline.egg-info`.
6. **Land the ONNX worktree to master**, then **implement the FLM/Q4NX stage** from its approved spec.
7. **Add the pre-upload quality gate and the GPU-memory preflight** — both small, both high daily payoff.
8. **Decide `logging_config.py`'s fate** (wire it in and delete the three `LogFn` triples, or delete it and drop structlog) and **resolve `datagen`/`gardener`** (restore or remove extras + references).
9. **Pin supply-chain fetches** (llama.cpp + Quark to commit SHAs; amd-quark/onnxruntime-genai to exact versions) and **make the REAP path configurable**.
10. **Add a CLI `_run()` timeout/watchdog** and clean up generated `_stage_*.py` scripts after success.
