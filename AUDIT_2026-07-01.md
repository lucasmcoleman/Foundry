# Foundry Audit — 2026-07-01

**Scope:** ML fine-tuning + hybrid-quantization pipeline orchestrator (ROCm / Strix Halo gfx1151), FastAPI + WebSocket UI, `pyproject` v0.3.0.
**Audited tree:** branch `feature/mq-hardware-items` @ `eb919af` **plus its uncommitted working-tree diff** (verified 2026-07-02). `master`, `feature/qat`, and `feature/rocmfpx-composition` all point at the same `eb919af` and are pushed to `origin`.
**Threat model:** single-user homelab; internet ingress behind SWAG + Authelia; primary interface is loopback-bound UI + CLI.

---

## Health grade: **B**

Genuinely strong engineering with a few sharp edges that need filing before the next commit and the next deploy. The June 2026 remediation is fully landed and verified (loopback-by-default + `hmac.compare_digest` auth, dataset normalizer, completion-marker resume, scoped HF token, service/entry-module consolidation). QAT and ROCmFPX shipped this cycle with real test coverage (225 offline tests green) and no injection or shell-exec regressions. What pulls the grade to B: (1) a **live, uncommitted wide-open CORS middleware** on the active branch that re-opens a shell-equivalent surface; (2) a **confirmed silent wrong-artifact upload** when ROCmFPX runs without MagicQuant; (3) the **container deployment path is LAN-exposed with no auth by default**, bypassing the very bind guard the June pass added; and (4) real merge debt (`feat/quark-onnx-stage` is a finished, unmerged, pre-hardening branch). None of these cause data loss or corruption, and all are scoped by the single-user threat model — but three of them are one `git commit` / `docker compose up` away from mattering.

## What's good

These are concrete, verified strengths — not boilerplate.

- **Injection stays closed in the new stage code.** `core/_rocmfpx_entry.py`, `core/_qat_entry.py`, `core/_magicquant_entry.py`, and their `services.py` builders pass config through JSON (`json.dumps` / `parse_config`), never f-string-interpolate untrusted strings into generated Python, and every `subprocess` call uses list args — **no `shell=True` anywhere** in `core/` or `ui/` (grep-verified).
- **June hardening is intact and uniform.** `ui/app.py` still carries `REQUIRE_AUTH` fail-closed (`:65`), `hmac.compare_digest` Bearer + WS-token auth (`:78`, `:1511`), and `select_host()` refusing a non-loopback bind without a key (`:1527`, applied at `:1551`). Every new `/api` route (QAT/ROCmFPX handlers, workflow save/load/delete) is gated behind `Depends(verify_api_key)` with no gaps.
- **HF token blast radius stays narrow.** The upload-stage-only token injection was correctly **not** extended to the new `do_qat`/`do_magicquant`/`do_rocmfpx` handlers — they call `run_script(...)` without `inject_hf_token=True`, so the new stages don't widen the credential's reach.
- **Real supply-chain controls.** Both live remote-code paths (ROCmFPX's `git clone` + build-script execution, MagicQuant's llama.cpp auto-install) pin to an explicit commit/tag, not a floating branch, with the rationale documented in-code (`_rocmfpx_entry.py:11-13`, `_magicquant_entry.py:11-13`). (Enforcement has a reuse gap — see F-6.)
- **The service+entry-module pattern scaled cleanly.** All 8 stages now have a `core/_<stage>_entry.py` (verified present), and QAT/ROCmFPX were added following the same JSON-config → importable-`run()` discipline, so CLI and UI share one code path per stage.
- **Test discipline paid off.** 225 offline tests pass (`-m "not gpu and not slow"`), up from 79 in June, including 58 ROCmFPX-specific tests covering the tricky translation/base-type logic. Entry modules keep pure-logic helpers stdlib-only so they're unit-testable without a GPU.
- **Comments read as real engineering notes.** `pick_base_type()`'s docstring cites the actual 69 GB "Q4" no-op bug it prevents; `docs/rocmfpx.md` and `CLAUDE.md` are grounded in incidents, not generic prose.
- **Prior packaging smell genuinely fixed.** MagicQuant/heretic-llm sibling repos moved to an optional `[external]` pyproject extra (commit `dba6a20`), so `pip install .` and the Docker build no longer require unpublished local repos.

## Findings

Ordered by severity. Each has a location and a suggested fix. "Confirmed" = adversarially verified against the current tree; "Minor/unverified" flagged as such.

### High

#### F-1 (security) — Wide-open CORS middleware on the live working tree re-opens the shell-equivalent `/api/run` surface
**`ui/app.py:83-90` (uncommitted diff on `feature/mq-hardware-items`)**

The active, un-committed working-tree diff adds:
```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```
`allow_origins=["*"]` + `allow_credentials=True` makes Starlette **reflect the caller's `Origin`** and permit credentialed cross-origin requests, and `allow_methods=["*"]`/`allow_headers=["*"]` lets any preflight through — applied globally, including to `POST /api/run`, which launches attacker-controllable Python subprocesses by design. Against a loopback bind this converts a "not reachable from the network" surface into a **drive-by target**: a malicious web page the operator visits can have their browser POST to `http://127.0.0.1:7865/api/run` and the now-permissive preflight lets it through. Foundry's auth is `Authorization`-header (not cookie) based, which limits classic credentialed-CSRF, but it does not close the loopback drive-by path, and this change is one `git commit` from becoming permanent with no review.
**Fix:** Do not commit this as-is. Restrict `allow_origins` to the actual UI origin(s) (`http://127.0.0.1:7865`, `http://localhost:7865`), drop the `allow_credentials` + wildcard combination, and scope `allow_methods`/`allow_headers` to what the SPA needs. If CORS isn't actually required (same-origin SPA), remove the middleware entirely.

#### F-2 (correctness / quality-ops) — Upload stage silently ships the wrong (huge, unquantized) GGUF when ROCmFPX runs without MagicQuant
**`core/hf_upload.py:101-113` (`discover_upload_files`); `core/_rocmfpx_entry.py:394` (`_ensure_bf16_gguf` cache path)**

`discover_upload_files()`'s GGUF branch globs only `<output>/magicquant/*.gguf`, then falls back to a single `<output>/model-bf16.gguf`; it **never globs `<output>/rocmfpx/*.gguf`** (the string `rocmfpx` does not appear in `hf_upload.py`, and `HFUploadConfig` has no `did_rocmfpx`/`upload_rocmfpx` flag). Meanwhile `_ensure_bf16_gguf` caches its BF16 conversion intermediate at **exactly that fallback path**, `out_dir / "model-bf16.gguf"`. So the supported, documented combination `--rocmfpx --no-magicquant --upload-to user/repo` (magicquant is on by default; disabling it is the trigger) leaves both the real deliverable (`rocmfpx/*.gguf`) and a 30–70 GB raw BF16 intermediate on disk — and the upload stage silently picks the intermediate, logs a normal "Found N files" success line, and publishes the massive **unquantized** model while the actual ROCmFPX artifacts stay local. `dry_run()` shares the same `discover_upload_files` call, so even the preflight preview doesn't catch it.

Verifier consensus: confirmed real and reachable from the CLI (3/3 on the quality-ops framing). The nuance worth stating honestly: `docs/rocmfpx.md:118-121` **does** document that "rocmfpx output isn't wired to upload yet" — but neither the docs nor the design spec anticipate the *silent substitution of the much larger BF16 intermediate*, which is the actual defect. This is worse than a missing feature: it's a silent wrong-output with a success log.

Note the uncommitted diff adds an identical `_ensure_bf16_gguf` to `core/_magicquant_entry.py` (measured-search path), which also writes `out_dir/model-bf16.gguf` — broadening the footprint of the exact file that becomes the accidental fallback.
**Fix:** Add a `rocmfpx/*.gguf` glob (or a dedicated `upload_rocmfpx` flag mirroring `did_magicquant`) to `discover_upload_files`, wired from `stage_upload`/`do_upload`. Make the BF16 fallback loud and opt-in — e.g. `"no magicquant/rocmfpx GGUF found; falling back to raw BF16 — this is NOT quantized"` — rather than a silent equal-priority candidate.

#### F-3 (security / ops) — Container deployment path is LAN-exposed with no auth by default, bypassing the June bind guard
**`Dockerfile:69` (CMD); `docker-compose.yml:12-13, :16`**

`Dockerfile` line 69 is `CMD ["uvicorn", "ui.app:app", "--host", "0.0.0.0", "--port", "7865"]`, invoking uvicorn **directly**. The loopback-by-default + fail-closed logic lives in `ui/app.py`'s `main()`/`select_host()` (`:1527-1551`) and in `ui/run.sh`, **neither of which runs under this CMD**. `docker-compose.yml` publishes `${FOUNDRY_UI_PORT:-7865}:7865` on all host interfaces with `FOUNDRY_API_KEY=${FOUNDRY_API_KEY:-}` empty by default; with no key set, `verify_api_key` is a no-op (`REQUIRE_AUTH` also unset by default). Net: `docker compose up` with defaults exposes the shell-equivalent `/api/run` UI to the entire LAN with zero credentials — the exact configuration the June hardening closed for the `python ui/app.py` path. The compose file has an in-file comment warning to set `FOUNDRY_API_KEY` (lines 9-11), which mitigates *awareness* but not the default behavior.
**Fix:** Make the container inherit the guard. Either change CMD to run via `ui/run.sh` (so `select_host()` applies), or set `FOUNDRY_UI_HOST=127.0.0.1` as the image default and require an explicit opt-in to `0.0.0.0`, and bind the compose port to `127.0.0.1:${FOUNDRY_UI_PORT}` by default. At minimum, refuse to start with `--host 0.0.0.0` and an empty `FOUNDRY_API_KEY`.

#### F-4 (security) — Unmerged `feat/quark-onnx-stage` worktree predates the June auth/bind hardening
**`.worktrees/quark-onnx/ui/app.py:39-45` (`verify_api_key`), `:1271` (WS), `:1287` (`uvicorn.run(..., host="0.0.0.0")`)**

The `feat/quark-onnx-stage` worktree (HEAD `da4c16a`, ~2026-05-07, 22 commits ahead of master, **not on any remote**) contains a pre-hardening `ui/app.py`: plain-string Bearer compare instead of `hmac.compare_digest`, no `REQUIRE_AUTH`, and a hardcoded `uvicorn.run(app, host="0.0.0.0", port=port)` with no `select_host()` guard. Smoke-testing this worktree's UI directly (`cd .worktrees/quark-onnx && python ui/app.py`) binds all interfaces with auth effectively disabled by default, exposing `/api/run` unauthenticated on the LAN.

Honest calibration: this branch **predates** the June-9 hardening rather than reverting it, and a normal three-way merge would keep master's hardened lines (the ONNX branch never touched them), so the regression does **not** auto-propagate on merge. The exposure is real but conditional on either (a) running this specific worktree's server, or (b) a pathological `-X theirs` merge. Two of three verifiers landed on High as a merge-blocker; one argued Low (stale relic). Treated as High because it's a live checkout inside the repo and the branch is described as having "final pre-merge fixes."
**Fix:** Before merging, rebase `feat/quark-onnx-stage` onto current `master` so it picks up `hmac.compare_digest`, `REQUIRE_AUTH`, and `select_host()`. Until then, do not run this worktree's UI directly; if you must, set `FOUNDRY_API_KEY` first and confirm it still binds `0.0.0.0` (it will) so it's only done on a trusted network. If the ONNX stage is abandoned, delete the worktree.

### Medium

#### F-5 (quality-ops) — UI pipeline runner has no per-stage timeout or hang watchdog, unlike the CLI
**`ui/app.py:278` (`run_script`); contrast `core/pipeline.py:253-330` (`_run`)**

`core/pipeline.py._run()` builds process-group-kill + a background drain thread *specifically* because "a gfx1151 kernel hang otherwise wedges the CLI indefinitely" (comment `:267-269`), and it's unit-tested (`tests/test_run_timeout.py`). The UI's separate runner, `run_script()`, has **no** timeout anywhere in its signature or body — `async for raw in proc.stdout` blocks forever if the child goes silent. `/health` (`:1279`) only asserts the event loop is alive, and `/api/state` exposes no last-log-activity timestamp a watchdog could use. Given this box's documented pattern of unattended overnight GPU runs, a wedged kernel op during a UI-launched run hangs silently until someone checks in the morning, with the healthcheck green the whole time. The only recovery is the manual `/api/stop` button (`:1484`).

Calibrated to Medium (down from an initial High): impact is availability/wasted-compute on a single-user box, a manual recovery path exists, and a *forced* default timeout would wrongly kill legitimate multi-hour training runs (which is why the CLI's `--stage-timeout` defaults to `None`). The June audit already logged this class as `[L-cli-timeout]`.
**Fix:** Surface a `last_log_activity` timestamp in `/api/state` so a stalled job is externally detectable, and add an **opt-in** watchdog (generous default, configurable) that mirrors `_run`'s process-group-kill for unattended runs.

#### F-6 (security / correctness) — Supply-chain pin (ROCmFPX + llama.cpp) is enforced only on first install; reuse of any existing checkout skips re-checkout
**`core/_rocmfpx_entry.py:177-198` (`find_rocmfpx`), `:314-334` (`ensure_rocmfpx` reuse branch); `core/_magicquant_entry.py:30-63`** *(consolidates two related observations)*

`ensure_rocmfpx()` calls `find_rocmfpx()` first and returns immediately if any working binary is discovered (env var, `~/ROCmFPX`, `./ROCmFPX`, a hardcoded sibling path) — none of the probes check the git commit. Even in the auto-install fallback, the reuse branch is `if (install_dir / ".git").exists(): print("Reusing existing checkout ...")` and skips straight to build — it never re-runs `git checkout ROCMFPX_PIN`. So only a from-scratch clone actually enforces the pin. A `git pull`, branch switch, or interrupted first clone leaves `~/ROCmFPX` (or `~/llama.cpp`) on an untested commit that every subsequent run trusts and builds with no warning — silently defeating the documented pin on every run after the first. `find_llamacpp`/`ensure_llamacpp` have the identical shape.
**Fix:** On the reuse path, `git rev-parse HEAD` and compare to `ROCMFPX_PIN`/`LLAMACPP_PIN`; on mismatch, re-checkout the pin (or fail loudly). Either enforce or explicitly document that hint/env/hardcoded discovery paths are operator-trusted and pin-exempt.

#### F-7 (architecture) — Three independent, occasionally-disagreeing copies of "source model" priority resolution
**`core/_rocmfpx_entry.py:351-378`, `core/_magicquant_entry.py:79-107`, `core/reap_common.py:137-152`; called from `core/pipeline.py:1072, 1156-1159`**

`_rocmfpx_entry.resolve_source()` is a near-verbatim copy of `_magicquant_entry.resolve_source()`, and both duplicate `reap_common.resolve_artifact_source()`. `stage_magicquant`/`stage_rocmfpx` resolve once in the parent using `artifacts.output_dir` — which is **never `.resolve()`'d**, so it stays relative when `--output-dir` is relative (the dataclass default is literally `./output`, and `scripts/run_agentworld_*.py` hardcode a relative `./output/...`). That relative path is then passed as `source_override` into a subprocess that always runs `cwd=PROJECT_ROOT`, which re-resolves it a *second* time. When the CLI is invoked with a relative `--output-dir` from any directory other than `PROJECT_ROOT` (cron/systemd/wrapper), both fallback candidates land on the wrong path and the stage aborts with "no source model found" even though the parent process resolved it fine. The common "cd into repo, then run" pattern self-heals by accident, masking the bug.
**Fix:** Delete the two entry-module copies and import `reap_common.resolve_artifact_source` directly; resolve `artifacts.output_dir` to an absolute path once in `Artifacts.__init__` (mirroring the UI's `_resolve_out`).

#### F-8 (architecture) — `core/config.py`'s "centralized" settings module is unused except for one field, and its duplicated defaults have silently drifted
**`core/config.py:22-86` (`FoundrySettings`); sole usage `ui/app.py:1541`**

The docstring instructs `from core.config import settings; print(settings.model_name)`, but the only real usage anywhere is `foundry_settings.ui_port` at `ui/app.py:1541` (itself redundant with pydantic-settings' env handling). None of the other ~29 fields reach `TrainingConfig`/`TrainingCfg`, and the duplicated defaults have already diverged: `config.py:38` sets `max_seq_length=8192` vs the real `4096`, and `:48` sets `optim='adamw_8bit'` vs the real `'paged_adamw_8bit'` (which matters on this box per `CLAUDE.md`'s ROCm/bitsandbytes notes). A contributor following the module's own documented pattern to change training behavior would silently affect nothing.
**Fix:** Either wire `FoundrySettings` in as the actual default source for the training configs (removing the duplicate field lists), or delete the ~29 unused fields and scope `core/config.py` to the `ui_port`/auth settings it actually drives.

#### F-9 (architecture) — Stale architecture docs: `FOUNDRY_MAP.md` describes a removed symlink/4-stage pipeline; `README.md` omits ROCmFPX
**`FOUNDRY_MAP.md` (whole file, dated 2026-04-03); `README.md:3, :144-168`**

`FOUNDRY_MAP.md` still states `Symlink: MagicQuant/magicquant -> ...` and lists only Training/Export/MagicQuant/Upload; that directory/symlink no longer exists (MagicQuant is an out-of-tree subprocess via the `[external]` extra). Unlike `HANDOFF.md`/`OPEN_QUESTIONS.md`, it never got a `> HISTORICAL` banner and reads as current. Separately, `README.md:144` says "The pipeline has seven stages" and describes Training/Export/Heretic/REAP/QAT/MagicQuant/Upload but **never mentions ROCmFPX**, even though `CLAUDE.md` gained a full ROCmFPX section in the same commit (`fc1fe4c`). A newcomer reading only `README.md` wouldn't learn the stage exists.
**Fix:** Add a `> HISTORICAL` banner to (or delete) `FOUNDRY_MAP.md`, and add a ROCmFPX subsection to `README.md`'s stage list to match `CLAUDE.md`.

#### F-10 (quality-ops) — `hf_upload.py` — the terminal, network-mutating, irreversible stage — has no dedicated tests
**`core/hf_upload.py` (833 lines); only reference is `tests/test_imports.py:36-38`**

Every other core stage has a dedicated test file, but the highest-blast-radius module (irreversible HF publish, `HF_TOKEN` handling, large binary transfer) has none. `discover_upload_files()`, `generate_model_card()`, `dry_run()`, and `upload()`'s retry/error paths are entirely untested — which is exactly what let F-2 ship unnoticed.
**Fix:** Add `tests/test_hf_upload.py` covering `discover_upload_files()` across all stage-output combinations (including rocmfpx-without-magicquant), model-card field coverage, and `upload()`/`dry_run()` error paths with a mocked `HfApi`.

#### F-11 (quality-ops) — BF16-GGUF conversion cache is existence-only, contradicting the project's own `markers.py` rationale
**`core/_rocmfpx_entry.py:394` (`_ensure_bf16_gguf`); replicated in the uncommitted `core/_magicquant_entry.py:110-135`**

`markers.py`'s docstring is explicit: "Existence-based skips are unreliable ... a crash before [the artifact] would false-pass an existence check." Yet `_ensure_bf16_gguf` does exactly `if cached.exists(): return str(cached)` — no size check, no marker, no GGUF-magic validation that the external `convert_hf_to_gguf.py` subprocess actually finished. A conversion killed mid-write (OOM/SIGKILL during an overnight run) leaves a truncated `model-bf16.gguf` that every later ROCmFPX/MagicQuant run treats as valid and reuses, producing corrupt quantized output far downstream. The uncommitted MagicQuant measured-search path adds a second copy of this smell.
**Fix:** Gate the BF16-GGUF cache behind a completion marker (`core/markers.py`), or at minimum a non-zero-size + GGUF-magic-byte check before trusting an existing file.

#### F-12 (correctness) — `feat/quark-onnx-stage` predates the entry-module consolidation and will conflict on merge
**`.worktrees/quark-onnx/core/services.py:958-1041`, `.worktrees/quark-onnx/core/pipeline.py:1295`**

The branch is missing the June consolidation commits, so there is no `core/_onnx_entry.py`; `OnnxService.build_script` generates its subprocess script by embedding raw Python as f-string text, including a hand-written 4th copy of the reap→heretic→merged resolver (the same duplication as F-7, minus test coverage). `git merge-tree` against `master` produces ~13 real conflict hunks across `CLAUDE.md`, `core/pipeline.py`, `core/services.py`, `ui/app.py`, `ui/index.html`, `pyproject.toml`. Landing as-is either ships a structurally-inconsistent 9th stage or requires nontrivial rework.
**Fix:** Rebase onto current `master` and port `OnnxService` onto the `_entry_shim` + `core/_onnx_entry.py` pattern before merging (see the "What's missing" roadmap).

A refresh/sleep/blip during a multi-hour run loses prior live-view text (the on-disk `.log` survives, so no data loss). **Fix:** buffer the last N lines (or tail the current stage's on-disk log) and replay to newly-connected WS clients.

### Note on the uncommitted `LLAMACPP_PIN` bump
The working-tree diff changes `LLAMACPP_PIN` from `b4585` to `gguf-v0.19.0` in both `core/_magicquant_entry.py:23` and `core/pipeline.py:342`. The new ref **does resolve** (it's a real annotated tag on `ggml-org/llama.cpp`, verified via `git ls-remote`), so the `--branch gguf-v0.19.0` clone will check out — but `gguf-v0.19.0` is a **gguf-py package release tag**, not a llama.cpp build-number tag like `b4585`, so it points at a different commit with different build/binary semantics. This is an unreviewed supply-chain change; verify the checked-out tree actually builds `llama-quantize`/`convert_hf_to_gguf.py` cleanly on gfx1151 before committing.

## What's missing — capability roadmap (ordered by value-to-effort)

1. **Merge `feat/quark-onnx-stage` (ONNX / amd-quark INT4-AWQ stage) into master — transformative, moderate.** QAT and ROCmFPX are already merged and pushed; the ONNX stage is the *only* finished-but-unmerged capability left (complete with UI panel, e2e smoke test, docs). Rebase onto `master` to pick up the entry-module pattern (F-12) and the June hardening (F-4), resolve the ~13 bounded stage-registration/doc conflicts by combining both additions, run the full suite + `tests/test_script_equivalence.py`, then rebuild/re-verify the Docker image and bump the stage count in `CLAUDE.md`.
2. **Pre-upload quality gate — high value, easy.** Still absent: `stage_upload_dry_run` validates credentials/repo/card but never model correctness, so a bad quant tier or corrupt hybrid GGUF ships straight to the Hub. Add a lightweight stage that loads each produced artifact, runs one short deterministic generation (catches load/dtype failures), and optionally computes a perplexity delta vs the BF16 baseline (llama.cpp is already vendored). Fail before `stage_upload` on hard failure; warn-and-continue past a configurable perplexity threshold. Independently flagged by the June audit and still open.
3. **FLM / Q4NX pipeline stage — transformative, moderate.** This is the box's *actual* NPU deployment target (FastFlowLM on the same Strix Halo XDNA2 NPU) and the one output format Foundry can't produce end-to-end. Design is Approved (`docs/superpowers/specs/2026-05-07-flm-q4nx-stage-design.md`) and the hard conversion logic already exists as a tested sibling repo (`/server/programming/FLM_Q4NX_Converter`). Implement as a near-clone of the ONNX stage pattern — **land the ONNX merge first**, since the spec names it the template.
4. **Run/experiment observability — medium, moderate.** `GET /api/runs` reports only filenames/sizes/mtimes; there's no way to compare runs without opening raw logs. Have the training/magicquant scripts emit a small `run_summary.json` (final loss, per-tier eval perplexity — MagicQuant already writes `search_results.json`, wall-clock per stage) and add a `/api/runs` summary table/sparkline. Additive, no schema migration.
5. **Fine-tune a small specialist for obsidian-brain's nightly LLM tasks — medium, moderate (exploratory).** obsidian-brain's nightly pipeline hits a general-purpose LAN llama-swap endpoint today. Foundry already produces exactly what llama-swap consumes (GGUF, and post-merge ROCmFPX-tuned GGUF for this box). Harvest obsidian-brain's historical classification/MOC/ledger outputs as `messages`-format training pairs (via the `training-data` repo's datagen pattern + Foundry's `dataset_format` normalizer), run a normal QLoRA on a small 1–4B base, and register the result in llama-swap. Net-new idea; mark exploratory.
